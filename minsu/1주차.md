# 😀 REST API

### 정의

REST (Representational State Transfer) API는 웹 기반 응용 프로그램의 통신 방식 중 하나로, 클라이언트와 서버 간의 통신에서 **REST 아키텍처** 원칙을 따르는 것을 말합니다.

REST API는 **HTTP 프로토콜**을 사용합니다. 클라이언트가 URI(Uniform Resource Identifier)를 통해 웹 리소스를 요청하고, 서버는 요청을 받아 JSON, XML 등의 형태로 응답합니다. 
간결하고 확장성이 좋으며, 웹 브라우저, 모바일 앱, 서버 등 다양한 클라이언트에서 사용할 수 있다는 장점이 있어 현재는 대부분의 웹 서비스에서 REST API를 사용하고 있습니다.


## REST API 설계 원칙

### URI는 정보의 자원을 나타내야 한다.

- URI(Uniform Resource Identifier)는 인터넷에 있는 자원을 나타내는 고유 식별자입니다. RESTful API에서는 URI를 사용하여 자원을 나타내기 때문에 URI는 정보의 자원을 정확하게 표현해야 합니다.

### 자원에 대한 행위는 HTTP Method(GET, POST, PUT, DELETE)로 표현한다.

- 자원에 대한 행위는 HTTP Method를 이용하여 표현합니다. HTTP Method는 클라이언트가 서버에게 요청하는 행위를 정의하는 방법으로, 가장 기본적인 HTTP Method는 GET, POST, PUT, DELETE가 있습니다.

### 메시지는 스스로를 설명할 수 있어야 한다.

- RESTful API에서는 메시지에 대한 메타데이터 정보를 포함하여 스스로를 설명할 수 있어야 한다. 이를 통해 클라이언트가 이해하기 쉬운 메시지를 전달할 수 있습니다.

### 메시지는 캐시 가능(cacheable)해야 한다.

- RESTful API에서는 메시지에 캐시 가능성을 포함시켜야 합니다. 이는 서버의 부하를 줄이고 성능을 향상시킬 수 있기 때문에 중요합니다.

### 하이퍼미디어 API를 제공해야 한다.

- RESTful API는 하이퍼미디어 API를 제공해야 합니다. 하이퍼미디어 API란 서버가 제공하는 API 문서를 통해 클라이언트가 서버와 상호작용할 수 있는 방법을 제공하는 것입니다. 이를 통해 클라이언트가 서버와 상호작용할 때 불필요한 문서 요청을 최소화하고, 효율적인 API 호출을 할 수 있습니다.

# 😀 HTTP/HTTP2.0/ HTTP3.0/ HTTPS

### HTTP

HTTP(Hypertext Transfer Protocol)는 인터넷에서 데이터를 주고받기 위한 통신규약(프로토콜) 중 하나입니다. 이 프로토콜은 클라이언트와 서버 사이에서 요청과 응답을 주고받는 방식으로 동작합니다.

### HTTP/1.0

초기 HTTP 버전으로, 클라이언트의 요청마다 새로운 TCP 커넥션을 열고, 한 번의 요청에 대해 한 번의 응답을 받는 방식으로 동작합니다. 이로 인해 많은 수의 요청과 응답이 발생하면 불필요한 TCP 커넥션의 생성과 종료가 반복되어 성능상의 문제가 발생할 수 있습니다.

### HTTP/1.1

HTTP/1.0의 문제를 해결하기 위해 도입된 버전으로, 한 개의 TCP 커넥션을 유지하고, 이를 통해 여러 개의 요청과 응답을 처리할 수 있도록 되어 있습니다. 이로 인해 성능이 향상되었습니다.

### HTTP/2.0

HTTP/1.1에서 발전된 버전으로, 한 개의 TCP 커넥션을 유지하면서도 다중 요청과 응답을 처리할 수 있도록 되어 있습니다. 이를 위해 요청과 응답을 바이너리 형식으로 인코딩하고, 헤더 압축 기능을 제공하며, 서버 푸시(Server Push)라는 새로운 기능을 추가했습니다.

### HTTP/3.0

HTTP/2.0에서 발전된 버전으로, UDP 기반의 QUIC(Quick UDP Internet Connections) 프로토콜을 이용해 통신합니다. UDP 기반의 프로토콜을 사용하면 데이터 전송에 걸리는 시간을 줄일 수 있어서 빠른 성능을 보장할 수 있습니다.

### HTTPS(Hypertext Transfer Protocol Secure)

HTTP 프로토콜의 보안 버전으로, 클라이언트와 서버 간에 데이터 전송 시 암호화를 적용하여 보안을 강화합니다. HTTPS는 SSL(Secure Socket Layer) 또는 TLS(Transport Layer Security) 프로토콜을 이용하여 데이터를 암호화하고, 인증서를 사용하여 상대방의 신원을 확인합니다. 이로 인해 중간자 공격 등의 보안 문제를 예방할 수 있습니다.

# 😀 스프링 프레임워크를 사용하는 이유

### 모듈성과 유연성

스프링은 필요한 모듈만 선택하여 사용할 수 있도록 모듈성이 뛰어나며, 개발자가 프로젝트 규모에 따라 더 유연하게 사용할 수 있습니다. 또한, 스프링은 다양한 데이터베이스와 연동할 수 있는 ORM(Object-Relational Mapping) 기능을 제공하여 개발자가 원하는 데이터베이스를 선택하여 사용할 수 있습니다.

### DI와 IoC

스프링은 의존성 주입(DI)과 제어 역행(IoC)이라는 디자인 패턴을 적극적으로 활용합니다. DI는 객체 간 의존성을 개발자가 직접 관리하는 것이 아니라, 스프링이 자동으로 관리하도록 도와주는 것입니다. IoC는 객체를 생성하고 관리하는 역할을 스프링이 수행하며, 개발자는 객체의 생성과 관리를 신경쓰지 않고 비즈니스 로직 개발에만 집중할 수 있습니다. 이는 코드의 가독성과 유지 보수성을 향상시키며, 개발 생산성을 높이는데 큰 도움을 줍니다.

### AOP(Aspect-Oriented Programming) 

스프링은 AOP를 적극적으로 활용하여 여러 모듈에 걸쳐 있는 공통 로직을 하나의 모듈로 분리하여 개발할 수 있게 도와줍니다. 이는 중복 코드를 제거하고, 유지 보수성과 확장성을 향상시킵니다. 예를 들어, 로그인 기능을 구현하는 코드가 여러 모듈에서 사용된다면, 이를 하나의 모듈로 분리하여 개발할 수 있습니다.

### 테스트 용이성

스프링은 JUnit과 같은 테스트 프레임워크와 연동이 용이하며, Mock 객체와 같은 도구를 제공하여 단위 테스트와 통합 테스트를 보다 쉽게 수행할 수 있게 도와줍니다. 이는 개발 생산성과 코드의 품질을 높이는 데 큰 도움을 줍니다.

### 보안 

스프링은 보안 기능을 내장하고 있어서, 인증, 권한 부여 및 암호화 등의 보안 요구 사항에 대응할 수 있습니다. 예를 들어, 스프링 시큐리티(Spring Security)라는 모듈을 사용하면 보안 기능을 쉽게 구현할 수 있습니다.
